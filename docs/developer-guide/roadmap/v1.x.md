# v1.x — Future directions

Features that are on the radar but not scheduled. Each needs a dedicated design pass
before it becomes a real milestone.

---

## Database table definitions

Describe your data model in `dot.yaml`, get a working backend.

```yaml
apps:
  api:
    database:
      tables:
        - name: users
          fields:
            - { name: id,    type: uuid,   primary: true }
            - { name: email, type: string, unique: true }
          relations:
            - { type: has_many, table: posts }
```

From this, dot would generate:
- SQL migration files (goose, migrate, alembic, etc.)
- Model structs / ORM models
- Repository interfaces + CRUD implementations
- REST or gRPC endpoints per table

This is Rails-level productivity. Large surface area — schema evolution, relation handling,
migration ordering — needs its own design doc before implementation starts.

## Web dashboard

A team-friendly web UI that produces the same Spec as the CLI.
Useful for onboarding non-CLI developers or for visualising a monorepo's structure.

No code generation logic changes — it is purely an input layer, like the TUI and MCP.

---

## Generator versioning and lockfile

Pin each installed generator to an exact version.
`dot.lock` analogous to `go.sum` or `package-lock.json`.
Ensures reproducible scaffolding across machines and over time.

---

## dot reinit

Re-run `dot init` on an existing project. Full conflict detection — every file that
has changed since the original init surfaces as a conflict for the developer to resolve.
Useful when upgrading a generator to a major new version.

---

## Multi-language monorepo generators

Official generators for TypeScript, Python, and Rust services inside a single `dot.yaml`
monorepo. Today only Go is officially supported.

---

## Generator testing framework

A test harness that makes it easy to write integration tests for generators:
apply a Spec, assert the exact files and content produced, assert no conflicts.
Currently generators are tested by hand. A first-class testing framework would
make the community generator ecosystem more reliable.
