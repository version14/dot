## Problem Statement

Starting a project today means one of three paths, all broken:

1. **Opinionated starters** (create-t3-app, create-react-app, Spring Initializr) — fast, but
   you spend hours removing what you don't need. Want Prisma but not tRPC? Too bad. The stack
   is the stack. Customization is manual surgery after the fact.

2. **GitHub template repos** — someone did the work, but you clone it and spend 30+ minutes
   filtering: deleting example code, changing configs, removing half the packages, figuring out
   what the author intended vs. what you want.

3. **From scratch** — full control, but 200 lines of boilerplate before you write a single
   line of business logic. CI config, Docker setup, linter, formatter, test scaffolding —
   all manual, every time.

The gap: no tool lets you describe exactly what you want (stack, modules, config) and gets out
of your way. Not just for new projects — also for adding features to existing ones.

---

## What dot is

dot is a **universal project companion**. Not a framework starter, not a template runner.

- You describe what you want. dot builds it.
- After creation, dot knows your project's architecture and gives you commands to manage it.
- Works for REST APIs, frontend apps, game dev, data analysis, CLI tools, monorepos, microservices.
- Extensible by the community: anyone can publish generators for new languages, frameworks, patterns.

## Input Layers (CLI is one of many)

dot separates the **input** (how you describe what you want) from the **engine** (how it generates).

```
┌─────────────────────────────────────────────────┐
│                  Input Layers                   │
│  ┌──────────┐  ┌───────────┐  ┌─────────────┐  │
│  │  CLI TUI │  │ dot.yaml  │  │  Dashboard  │  │
│  │ (v0.1)   │  │ PaC (vC)  │  │ (web, vD)   │  │
│  └──────────┘  └───────────┘  └─────────────┘  │
│                ┌───────────┐                    │
│                │    MCP    │                    │
│                │ (future)  │                    │
│                └───────────┘                    │
└──────────────────────┬──────────────────────────┘
                       │
                     Spec
                       │
              ┌────────────────┐
              │ Generator Engine│
              │  (core, stable) │
              └────────┬───────┘
                       │
              ┌────────────────┐
              │Generator Registry│
              │ official + community│
              └────────┬───────┘
                       │
              ┌────────────────┐
              │ FileOp Pipeline │
              │(collect→resolve│
              │   →write)       │
              └────────┬───────┘
                       │
              ┌────────────────┐
              │  Project on disk│
              │+ .dot/config.json│
              └────────────────┘
```

**Input layers produce the same Spec. The engine doesn't know or care which input was used.**

- **CLI TUI** — interactive, ships in v0.1
- **dot.yaml (PaC)** — declarative, ships in Approach C
- **Dashboard** — web UI for teams, future
- **MCP** — AI agents scaffold via MCP protocol, future

## Use Cases (dot is not API-only)

dot generates any project type. The generator registry handles the specifics.

| Domain | Examples |
|---|---|
| Backend API | Go REST, Go gRPC, Python FastAPI, Node Express |
| Frontend | Next.js, React+Vite, SvelteKit, Vue |
| Full-stack | Next.js + Postgres, SvelteKit + SQLite |
| CLI tools | Go CLI (cobra/bubbletea), Python CLI (click/typer) |
| Libraries | Go library, Python package, npm package |
| Game dev | Godot (GDScript structure), Unity project layout |
| Data / ML | Python data science (notebooks + src), MLflow |
| Monorepo | Multiple apps + shared packages (see Approach C) |
| Microservices | Multiple services in a monorepo with per-service config |

The generator for "Go REST API" is just one generator in the registry. Adding "Godot project"
means adding a Godot generator. The engine doesn't change.

## Community Plugins / Generators

The `Generator` interface is public and documented. Anyone can:
- Create a generator for a new language or framework
- Publish it as a Go module (or binary, TBD)
- Register it with dot via config or CLI command

Examples of community generators:
- `dot-generator-fastapi` — Python FastAPI with SQLAlchemy
- `dot-generator-godot` — Godot 4 project layout
- `dot-generator-terraform` — Terraform project with module structure
- `dot-generator-dbt` — dbt data project with source/staging/mart structure

Community generators follow the same `Generator` interface as official generators.
There is no special registry needed in v0.1 — generators can be loaded from local paths.
A public registry is deferred to v1.0+ (like Terraform's registry or npm).

## Repository Structure — Progressive Split

Everything starts in `version14/dot`. Repos are extracted when a concrete trigger is hit,
not on a schedule. The internal package structure is designed for extraction from day one —
splitting is a module rename + import path find-replace, not a rewrite.

**The one rule that makes splitting easy:** nothing in `internal/` ever imports from `cmd/`.
`cmd/dot` depends on `internal/`. Never the reverse. Go enforces this.

---

### Phase 1 — v0.1 to v0.5: Everything in `version14/dot`

```
version14/dot/
├── cmd/dot/                  ← CLI entry point (thin: parse flags → call internal → print)
├── internal/
│   ├── spec/                 ← Spec, ProjectSpec, CoreConfig, ModuleSpec
│   │   └── yaml.go           ← FromYAML / ToYAML / Diff (ready for dot-core extraction)
│   ├── generator/            ← Generator interface, Registry, FileOp, CommandDef
│   ├── project/              ← ProjectContext, Load, Save (named "project" not "context" to avoid shadowing stdlib)
│   └── pipeline/             ← FileOp collect → resolve → write
├── generators/
│   ├── go/                   ← official Go generators
│   └── common/               ← language-agnostic (CI, Docker, etc.)
└── templates/                ← embedded via go:embed
```

---

### Phase 2 — Trigger: second consumer (Dashboard or MCP) needs core types

Extract `dot-core`. Migration is mechanical:
1. Create `version14/dot-core`
2. Copy `internal/spec`, `internal/generator`, `internal/project`, `internal/pipeline`
3. Update `go.mod` module name to `github.com/version14/dot-core`
4. In `version14/dot`: find-replace `github.com/version14/dot/internal/` → `github.com/version14/dot-core/`
5. `go get github.com/version14/dot-core` — done

```
version14/dot-core            ← Spec, Generator interface, FileOp, Registry,
                                ProjectContext, FromYAML, ToYAML, Diff
version14/dot                 ← CLI binary (imports dot-core + dot-std)
version14/dot-mcp             ← MCP server (imports dot-core, reads dot.yaml same as CLI)
```

PaC lives in `dot-core/spec/yaml.go`. The CLI and MCP both call `spec.FromYAML()`.
`dot plan` / `dot apply` are CLI commands — thin wrappers that call `spec.Diff()` from core
and format the output.

---

### Phase 3 — Trigger: community generators need a stable interface to depend on

Extract `dot-std`. Official generators move out of the CLI repo.

```
version14/dot-core            ← core engine (stable, versioned independently)
version14/dot-std             ← official generators (imports dot-core)
version14/dot                 ← CLI binary (imports dot-core + dot-std)
```

Community generators: `anyuser/dot-generator-fastapi` implements the `dot-core` Generator
interface. Loaded from local path in v0.5, from registry in v1.0+.

### Phase 4 — Trigger: community generator volume justifies discovery infrastructure

```
version14/dot-core            ← core engine
version14/dot-std             ← official generators
version14/dot-registry        ← plugin index (like Homebrew tap or Terraform registry)
version14/dot                 ← CLI binary
version14/dot-mcp             ← MCP server (if built)
```

## Core Types

Typed core + extension map. Official generators use `CoreConfig` (typed, IDE autocomplete).
Community generators use `Extensions` (free-form map, no core changes needed).

```go
// internal/spec/spec.go

type ProjectType string
const (
    ProjectTypeAPI      ProjectType = "api"
    ProjectTypeCLI      ProjectType = "cli"
    ProjectTypeMonorepo ProjectType = "monorepo"
    ProjectTypeLibrary  ProjectType = "library"
    ProjectTypeFrontend ProjectType = "frontend"
    ProjectTypeWorker   ProjectType = "worker"
)

type ProjectSpec struct {
    Name     string      `json:"name"`
    Language string      `json:"language"` // open string: "go", "python", "typescript", ...
    Type     ProjectType `json:"type"`
}

type Spec struct {
    Project    ProjectSpec    `json:"project"`
    Modules    []ModuleSpec   `json:"modules"`
    Config     CoreConfig     `json:"config"`              // official generators use this
    Extensions map[string]any `json:"extensions,omitempty"` // community generators use this
}

type CoreConfig struct {
    Linter     string `json:"linter"`     // "golangci-lint" | "eslint" | "none"
    Formatter  string `json:"formatter"`  // "gofmt" | "goimports" | "prettier" | "none"
    CI         string `json:"ci"`         // "github-actions" | "gitlab-ci" | "none"
    Deployment string `json:"deployment"` // "docker" | "docker-compose" | "vercel" | "none"
    Monitoring string `json:"monitoring"` // "grafana" | "datadog" | "none"
    Tracking   string `json:"tracking"`   // "posthog" | "sentry" | "none"
}

type ModuleSpec struct {
    Name   string         `json:"name"`
    Config map[string]any `json:"config,omitempty"` // module-specific config
}
```

Official generator reads: `spec.Config.Linter`
Community generator reads: `spec.Extensions["my-plugin-key"]`

## FileOp Pipeline (unchanged from previous version)

```go
// internal/generator/fileop.go

type FileOpKind int
const (
    Create   FileOpKind = iota
    Template
    Append
    Patch
)

// NOTE — subprocess model: When community generators run as subprocesses (v0.5+),
// Spec and []FileOp cross a process boundary as JSON. FileOpKind is currently an
// int iota, which serializes as 0/1/2/3 — fragile if constants are reordered or
// inserted. Before finalizing the subprocess protocol, change FileOpKind to a
// string type: type FileOpKind string; const (Create FileOpKind = "create"; ...).
// The Generator interface itself (Apply, RunAction) maps cleanly to JSON-RPC or
// stdin/stdout newline-delimited JSON — no interface change needed, just serialization.

const (
    AnchorImportBlock = "import_block"
    AnchorMainFunc    = "main_func"
    AnchorInitFunc    = "init_func"
)

type FileOp struct {
    Kind      FileOpKind
    Path      string
    Content   string
    Anchor    string
    Generator string
    Priority  int
}
```

**v0.1 Patch strategy — line-matching insert:**
- `AnchorImportBlock`: handles single-line imports (rewrites to block), no-import files (inserts block after `package`), and duplicate imports (skipped, not an error)
- `AnchorMainFunc` / `AnchorInitFunc`: brace-depth tracking to find closing `}`

**Conflict resolution:** Ops sorted by Priority (descending), then generator name (alphabetical).
Append/Patch: all applied in order. Create: highest Priority wins. Two generators at Priority=0
on the same file → abort with error naming both generators.

**Error handling:** All ops collected in memory before any write. Any failure → abort, no partial writes.

**Template format:** Go `text/template`, files embedded via `//go:embed templates`.

## Generator Interface, Registry, and ProjectContext

```go
// internal/generator/generator.go

type Generator interface {
    Name()      string    // unique identifier, e.g. "go-rest-api"
    Language()  string    // "go", "python", "*" for language-agnostic
    Modules()   []string  // module names this generator handles, e.g. ["rest-api"]
    Apply(spec Spec) ([]FileOp, error)          // called during dot init
    Commands()  []CommandDef                    // commands this generator registers
    RunAction(action string, args []string, spec Spec) ([]FileOp, error) // called by dot new/add
}

type CommandDef struct {
    Name        string   // "new route" — key in .dot/config.json available_commands
                         // Format: "verb noun" where noun is a single hyphenated word.
                         // No spaces in the noun: use "new rest-api", not "new rest api".
                         // Lookup: dot new <type> <args...> → key = "new " + type
    Args        []string // ["<name>"] — for dot help display only
    Description string
    Action      string   // "rest-api.new-route" — passed to RunAction
    Generator   string   // matches Generator.Name()
}

// Registry holds registered generators and resolves commands.
type Registry struct { generators []Generator }

// Register adds a generator to the registry.
// Returns an error if another already-registered generator claims the same
// (language, module) pair — e.g., two generators both listing "rest-api" for "go".
// Language "*" (language-agnostic) conflicts with any language-specific generator
// claiming the same module name.
// This catches generator conflicts at registration time, not at file-write time.
func (r *Registry) Register(g Generator) error

// ForSpec returns generators matching spec's language and requested modules.
// Matching rule: generator.Language() == spec.Project.Language || generator.Language() == "*"
// AND at least one of generator.Modules() appears in spec.Modules[].Name
func (r *Registry) ForSpec(spec Spec) []Generator

// Required test spec — internal/generator/registry_test.go:
//
// | Case                                | Registered generators        | Spec language | Spec modules      | Expected result           |
// |-------------------------------------|------------------------------|---------------|-------------------|---------------------------|
// | Exact language + module match       | go-rest-api (go, [rest-api]) | go            | [rest-api]        | [go-rest-api]             |
// | Language mismatch                   | go-rest-api (go, [rest-api]) | python        | [rest-api]        | []                        |
// | Language-agnostic matches any lang  | ci-gen (*, [ci])             | go            | [ci]              | [ci-gen]                  |
// | Language-agnostic + specific both match | go-rest-api + ci-gen    | go            | [rest-api, ci]    | [go-rest-api, ci-gen]     |
// | Module not requested                | go-rest-api (go, [rest-api]) | go            | [postgres]        | []                        |
// | Multiple modules, partial match     | go-rest-api (go, [rest-api, postgres]) | go | [rest-api]   | [go-rest-api]             |
// | Empty registry                      | (none)                       | go            | [rest-api]        | []                        |
// | Empty spec modules                  | go-rest-api (go, [rest-api]) | go            | []                | []                        |

// CommandsForSpec returns all CommandDefs from matched generators.
// Written to .dot/config.json as available_commands after dot init.
func (r *Registry) CommandsForSpec(spec Spec) []CommandDef

// Get returns a generator by Name(). Used during dot new dispatch.
func (r *Registry) Get(name string) (Generator, bool)
```

```go
// internal/project/project_context.go

// ProjectContext is written to .dot/config.json after dot init.
// It stores the project's full spec and all available post-creation commands.

type ProjectContext struct {
    DotVersion  string                `json:"dot_version"`        // semver of dot that created this
    SpecVersion int                   `json:"spec_version"`       // schema version; v0.1 = 1
    Spec        Spec                  `json:"spec"`               // authoritative project state
    Commands    map[string]CommandRef `json:"available_commands"` // keyed by CommandDef.Name
}

// CommandRef is the persisted form of a CommandDef (without display-only fields).
type CommandRef struct {
    Generator string `json:"generator"` // matches Generator.Name()
    Action    string `json:"action"`    // passed to generator.RunAction()
}

// Load reads .dot/config.json from the nearest git root.
// Returns an error if not inside a dot project.
func Load(startDir string) (*ProjectContext, error)

// Save writes the ProjectContext to .dot/config.json in the given project root.
func Save(root string, ctx *ProjectContext) error
```

**`dot new` dispatch flow:**
`dot new route UserController` →
1. `context.Load(".")` — finds `.dot/config.json` at nearest git root
2. Parse: key = "new route", args = ["UserController"]
3. `ctx.Commands["new route"]` → `CommandRef{Generator: "go-rest-api", Action: "rest-api.new-route"}`
4. `registry.Get("go-rest-api")` → generator
5. `generator.RunAction("rest-api.new-route", ["UserController"], ctx.Spec)` → `[]FileOp`
6. FileOp pipeline → write files

## Entry Commands

```
dot init                  # launch TUI → Spec → generators → write project
dot new <type> <name>     # generate new artifact in current project
dot add module <name>     # (v0.5) add module to existing dot project
dot help                  # list available commands from .dot/config.json
dot version               # print version
```

All commands except `dot init` traverse up from `$PWD` to git root for `.dot/config.json`.
If none found: `not a dot project (no .dot/config.json found)`.

**`dot init` idempotency:** If `.dot/config.json` already exists in the current directory,
`dot init` exits immediately with:
```
already a dot project — use dot add module to extend it
```
No files are written. No TUI is launched. Use `dot add module` to extend an existing project.
A future `dot reinit` command (v0.5+) can handle re-initialization with conflict detection.

## Conflict Resolution for Existing Projects

When `dot add module redis` modifies a file the user has already changed, dot writes
conflict markers directly into the file — same format as git merge conflicts. The user
resolves them in their editor, exactly like resolving a git conflict.

```go
// Example: dot add module postgres modifies an already-edited main.go

<<<<<<< dot (postgres generator)
db, err := sql.Open("postgres", os.Getenv("DATABASE_URL"))
if err != nil {
    log.Fatal(err)
}
=======
// user's existing code in this region
logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))
>>>>>>> current
```

**How it works:**
1. `.dot/manifest.json` stores the hash of every file at generation time
2. On `dot add module`, for each FileOp targeting an existing file:
   - If file hash matches manifest → file is unmodified → apply op directly
   - If file hash differs → file was modified by user → write conflict markers
3. `dot status` lists files with unresolved conflicts (like `git status`)
4. User resolves conflicts in editor, runs `dot resolve` to mark them done
5. `dot resolve` updates `.dot/manifest.json` with new hashes

**`.dot/` directory must be committed to git.** It is project state, not build output.
Gitignoring `.dot/` breaks `dot new` (no commands), `dot add module` (no manifest for
conflict detection), and any CI that uses dot commands. Generated project files are
committed normally — dot scaffolds, git tracks.

**`.dot/manifest.json`** — written by `dot init`, updated by `dot resolve`:
```json
{
  "files": {
    "main.go":            { "hash": "sha256:abc...", "generator": "go-rest-api" },
    "go.mod":             { "hash": "sha256:def...", "generator": "go-rest-api" },
    "docker-compose.yml": { "hash": "sha256:ghi...", "generator": "docker" }
  }
}
```

**This must ship in v0.1** (low cost, just hash + write on init). Without it, v0.5
`dot add module` has no way to detect user modifications.

**Exact conflict marker format and `dot resolve` UX:** implementation detail, decided
with collaborators before v0.5.

## Approach C — Project as Code (complex projects)

PaC for complex projects needs to describe more than a single app. It must support:
- **Monolith** — single app, all modules in one codebase
- **Monorepo** — multiple apps sharing a repo, with shared packages
- **Microservices in a monorepo** — each service independently deployable, own config

### Full PaC schema (proposed)

```yaml
# dot.yaml

meta:
  name: my-platform     # top-level project name
  type: monorepo        # monolith | monorepo | microservices

# Defaults inherited by all apps unless overridden
defaults:
  config:
    ci: github-actions
    linter: golangci-lint

apps:
  api-gateway:
    language: go
    type: rest-api
    modules:
      - rest-api
      - auth-jwt
      - redis
    config:
      deployment: docker
      monitoring: grafana
      tracking: sentry

  user-service:
    language: go
    type: rest-api
    modules:
      - grpc
      - postgres
    config:
      deployment: docker
      # inherits ci and linter from defaults

  frontend:
    language: typescript
    type: nextjs
    modules:
      - shadcn-ui
      - tanstack-query
    config:
      linter: eslint      # overrides default
      deployment: vercel
      tracking: posthog

  analytics-worker:
    language: python
    type: cli
    modules:
      - postgres
      - celery
    config:
      deployment: docker

shared:
  packages:
    - name: ui-components
      language: typescript
    - name: shared-types
      language: typescript
```

**`dot plan` diff unit:** module list per app. Adding `redis` to `user-service` in `dot.yaml`
when it's not in `.dot/config.json` → "will add: user-service/redis".

**`dot apply`** runs the delta: resolves added/removed apps/modules, runs generators, writes files.

### Database table definitions (proposed — future direction)

This is a significant scope addition. Flagged here as a design direction, not v0.1-v0.5 scope.

```yaml
apps:
  user-service:
    language: go
    type: rest-api
    modules: [grpc, postgres]
    database:
      tables:
        - name: users
          fields:
            - { name: id,         type: uuid,      primary: true }
            - { name: email,      type: string,    unique: true, index: true }
            - { name: created_at, type: timestamp, auto: true }
          relations:
            - { type: has_many, table: posts, foreign_key: user_id }

        - name: posts
          fields:
            - { name: id,      type: uuid,   primary: true }
            - { name: user_id, type: uuid,   index: true }
            - { name: body,    type: text }
```

From this, dot could generate:
- SQL migration files (with goose, migrate, alembic, etc.)
- Model structs / ORM models
- Repository interfaces + CRUD implementations
- API endpoints for each table

This is Rails-level productivity. "Describe your data model, get a working backend."
**Deferred to v2.0.** It's a large surface (schema evolution, relation handling, migration ordering)
and needs a dedicated design pass. The module spec (`ModuleSpec.Config`) is already designed
to hold table definitions when ready.

## Reviewer Concerns (unresolved after 3 review rounds)

These are real issues that need resolution before the affected features are built.
They do not block v0.1 core work but must be addressed before the features that depend on them.

1. **Community plugin loading mechanism (blocks v0.5 community generator support)**
   Go has no simple plugin system: `plugin.Open` is Linux/macOS only and fragile. Options:
   - **In-process:** Community generators are Go modules imported at compile time.
     Users compile a custom `dot` binary with their generators registered. Simple but requires
     recompiling dot to add generators.
   - **Subprocess/RPC:** Community generators are separate binaries. dot spawns them and
     communicates via stdin/stdout (like Terraform providers or Language Server Protocol).
     Flexible but more complex to implement.
   - **Embedded registry:** dot fetches generator binaries from a registry URL and runs them
     as subprocesses. Like Terraform's provider registry model.
   **Decision needed before v0.5.** The Generator interface is compatible with all three —
   the choice affects how generators are distributed and loaded, not the interface contract.

2. **Import-patch edge cases (affects v0.1 Go generators)**
   The v0.1 line-matching Patch for `AnchorImportBlock` does not handle: grouped blank imports
   (`import _ "pkg"`), build-tag-gated imports, or multiline `import` without parentheses
   (rare but valid Go). The known-safe subset is: standard `import ( ... )` blocks and
   single `import "pkg"` lines. Generators should only emit `AnchorImportBlock` Patch ops for
   these forms. Document this constraint in the generator authoring guide.

   **Required test spec — `internal/pipeline/patch_test.go`:**
   Implement as a table-driven test alongside the patch code. All cases must pass before
